# SGLang AMD Patches (v0.5.10)

Applied to SGLang 0.5.10 (`sglang-miles` branch) via `git apply --3way` in the Dockerfile.

## sglang.patch

### What it does

1. **Custom AllReduce on AMD**: Enables ROCm-compatible custom all-reduce for tensor parallelism.

2. **aiter CK MoE GEMM guard** (`unquant.py`):
   - `process_weights_after_loading`: Added `inter_dim % 64 != 0` guard before shuffling
     weights into the aiter CK MoE layout. The aiter CK stage1 GEMM kernel (`ck_moe_stage1`)
     requires `inter_dim % 64 == 0`. Without this guard, models with incompatible shapes
     (e.g. Qwen3-30B-A3B: `moe_intermediate_size=768`, `inter_dim=768/tp=8=96`) get their
     weights corrupted at load time, causing silent accuracy degradation.
   - `forward_cuda`: Same guard to fall back to Triton MoE kernel at runtime.
   - **Upstream potential**: Yes — this guard is a correctness fix that benefits all backends.
     It should be upstreamed to SGLang directly so that `SGLANG_USE_AITER=1` is safe for
     all model shapes without requiring users to manually disable aiter.

3. **Pre-tuned Triton MoE config** (`E=128,N=96,device_name=AMD_Instinct_MI300X_OAM.json`):
   - Baked into the image for Qwen3-30B-A3B on MI300X (E=128 experts, N=96 at TP=8).
   - Generated by sweeping 1600 configs with `tuning_fused_moe_triton.py`.
   - Gives ~+3% rollout throughput vs the default Triton config.
   - **Upstream potential**: Yes — can be contributed to the SGLang Triton MoE config repo
     once the device-name lookup is stable (currently `torch.cuda.get_device_name()` returns
     `""` in this ROCm environment, so both empty-name and OAM-name files are needed).
